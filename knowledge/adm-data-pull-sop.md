# Admonitor 拉数 SOP

> 每次拉数前必读本文件。不读不开始。

---

## 标准流程

### Step 1：获取 Token
- 先请求 Admonitor API Token，不能跳过
- Token 有效期注意，过期要重新获取

### Step 2：与小胡确认拉数参数
在实际拉数前，**必须逐项确认以下内容**：

| 参数 | 说明 |
|------|------|
| 指标 | 要拉哪些指标（曝光/点击/SIVT/GIVT等） |
| 维度 | 按什么维度切分（媒体/广告主/日期/地域等） |
| 客户名称 | 确认客户全称，注意拼写和系统内名称一致 |
| 时间范围 | 起止日期，注意时区 |
| 过滤条件 | 是否有特殊过滤（如只看OTT、只看某广告活动） |
| 数据粒度 | 天级/小时级/总量 |

**不确认直接拉 = 返工风险。**

### Step 3：执行拉数
- 按确认后的参数执行
- 记录拉数时间、参数快照

### Step 4：结果核验
- 总量是否合理（量级对不对）
- 关键指标有没有异常值
- 有疑问先问，不要直接交付

## 已知踩坑

| 坑 | 原因 | 避免方法 |
|----|------|----------|
| 指标口径不一致 | IGRP四舍五入规则、频次比例字段格式各项目不同 | 拉数前确认口径，Hankel case 的格式已固化 |
| 客户名称拼写错误 | 系统内名称与常用简称不同 | Step 2 必须确认系统内全称 |
| 时间范围偏差 | 时区、边界日期（含还是不含）不清楚 | 明确起止日期，确认是否包含边界 |

## 通识知识

- **GIVT**：通用无效流量（机器人、爬虫等）
- **SIVT**：复杂无效流量（人工刷量、真机兼职等）
- **布点过滤**：DR阶段按排期数据库过滤，只保留"点位×上线日期"有排期的数据
- **ETL → DM → DR**：三层计算链路，布点过滤发生在 DR 层
- **Gross / Net / Total Net**：详见 `knowledge/admonitor-arch.md`
