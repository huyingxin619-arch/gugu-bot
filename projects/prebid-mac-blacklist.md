# Prebid MAC黑名单升级

> 项目状态：方法论成型，测试数据已出，待分析 MAC vs IEEE 重合度
> 主要负责人：心宇（产品同事）

---

## 背景

秒针 Prebid 服务（尤其 OTT 端）原本不支持 ID/MAC 黑名单查询，原因是人力排期，非技术瓶颈。
目标：构建 MAC 黑名单查询方法论并评估是否上线。

---

## 现有 Prebid 过滤链（顺序）

1. **IP 黑名单**（秒针）
2. **威胁猎人**（第三方）
3. **IEEE 接口**（秒针）

---

## 评分机制

- 分值范围：**-1 ~ 10**
  - `-1` = 不认识（未收录）
  - `1~10` = 有风险，分越高风险等级越高
- 当前客户卡分：**9分**（过滤9分和10分的流量）

## MAC 黑名单插入方案

与 IP 黑名单并列，流量方查询时同时返回 IP 分和 ID 分，**取最高分**作为最终返回值。

---

## 测试数据（分母 = 当天进入 Prebid 接口总 PV）

| 维度 | 数值 |
|------|------|
| MAC 黑名单过滤量 | ~7% |
| IP 黑名单过滤量 | ~2% |
| IP 与 MAC 重合度 | ~1% |
| MAC 独立净增量 | ~5% |
| MAC 与威胁猎人重合度 | 不高（好消息）|

---

## 关键判断：值不值得上线

### 核心问题
**MAC 黑名单与 IEEE 过滤的重合度如何？**
- 重合度高 → 上线意义不大，量只是时间早晚的问题
- 重合度低 → 有独立增量价值，推进上线

### 其他判断维度
1. 分天分媒体的数据，是否有偏重/倚重
2. 前端业务同事意见 + 客户接受程度
3. 上线后对后验（SIVT 比例监测）的影响（⚠️ 容易被忽略）

> **后验注意：** Prebid 提前过滤的流量从后验 SIVT 监测视角会"消失"，不提前设计对比方案，上线后 SIVT 变化将无法解释清楚。
